title: "AI learning plans"
description: "How Kairo generates personalized 4-week study plans using your academic data and soft skills profile."

A learning plan is a 20-day (4-week) personalized study program generated by Kairo's AI for each coder. Every plan contains daily activities organized by week, a targeted soft skill focus, and a priority level across 6 focus cards — 2 high, 2 medium, and 2 low.

## Inputs used to generate a plan

Kairo pulls from three data sources before invoking the LLM:

<CardGroup cols={3}>
  <Card title="Moodle progress" icon="graduation-cap">
    `average_score`, `current_week`, and `struggling_topics` from the `moodle_progress` table
  </Card>
  <Card title="Soft skills profile" icon="brain">
    All five skill scores plus `learning_style` from `soft_skills_assessment`
  </Card>
  <Card title="Module context" icon="book">
    Module name, total weeks, and topic list from the `modules` and `topics` tables
  </Card>
</CardGroup>

## Plan types

Kairo supports two plan types, selected when requesting a new plan:

| Type | When to use | Extra inputs |
|------|-------------|-------------|
| `interpretive` | New coders or coders without Moodle data | Soft skills + module topics only |
| `analytical` | Coders with existing Moodle scores | Adds `average_score`, `struggling_topics`, `weeks_completed` |

<Note>
For `analytical` plans, if you don't explicitly send `average_score` and `weeks_completed` in the request body, the Python service auto-fetches them from `moodle_progress` for your `coder_id`.
</Note>

## How to request a plan

You trigger plan generation through the Node.js API, which fires an async request to the Python service:

<Steps>
  <Step title="Call the Node.js endpoint">
    Send a `POST` request to `/api/coder/plan/request`. You can optionally pass `plan_type` and `current_week`:

    ```javascript
    POST /api/coder/plan/request
    Content-Type: application/json

    {
      "plan_type": "analytical",
      "current_week": 3
    }
    ```

    The endpoint responds immediately with `{ "requested": true }` — plan generation happens asynchronously.
  </Step>
  <Step title="Node.js proxies to the Python AI service">
    Node.js calls `POST /generate-plan` on the Python FastAPI service (at `PYTHON_API_URL`), passing `coder_id`, `module_id`, `plan_type`, and `current_week`.
  </Step>
  <Step title="Python deactivates the previous plan">
    Before generating, the service calls `db_manager.deactivate_plans(coder_id)` to set `is_active = false` on all existing plans for that coder.
  </Step>
  <Step title="AI generates structured JSON">
    The Groq LLM (`llama-3.3-70b-versatile`) builds the full plan JSON with weeks, daily activities, and a `targeted_soft_skill`.
  </Step>
  <Step title="Plan is persisted to the database">
    The plan is saved to `complementary_plans` and individual activities are written to `plan_activities`.
  </Step>
</Steps>

## Plan generation for focus cards

Focus cards are generated separately via `POST /generate-focus-cards` on the Python service (called by Node.js). Each request produces exactly 6 cards:

```json
{
  "cards": [
    {
      "priority": "high",
      "topic": "SQL JOINs",
      "title": "Master INNER and LEFT JOINs",
      "theory": "A JOIN combines rows from two tables...",
      "practice": {
        "description": "Write 3 queries using both JOIN types",
        "example": "SELECT u.name, o.total FROM users u INNER JOIN orders o ON u.id = o.user_id;"
      },
      "pro_tips": ["Always alias your tables", "Check for NULLs when using LEFT JOIN"]
    }
  ]
}
```

The 6 cards always follow the distribution: **2 high**, **2 medium**, **2 low** priority. High-priority cards directly target your `struggling_topics`.

## Plan structure in the database

### `complementary_plans` table

```sql
CREATE TABLE complementary_plans (
  id                     SERIAL PRIMARY KEY,
  coder_id               INT NOT NULL,
  module_id              INT NOT NULL,
  plan_content           JSONB NOT NULL,         -- Full AI-generated plan JSON
  priority_level         priority_level_enum DEFAULT 'medium',  -- 'low' | 'medium' | 'high'
  soft_skills_snapshot   JSONB,                  -- Soft skills at generation time
  moodle_status_snapshot JSONB,                  -- Moodle data at generation time
  is_active              BOOLEAN DEFAULT TRUE,   -- Only one active plan per coder
  generated_at           TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  targeted_soft_skill    VARCHAR                 -- The skill this plan focuses on
);
```

### `plan_activities` table

Each day in a plan has one or more activities stored as individual rows:

```sql
CREATE TABLE plan_activities (
  id                     SERIAL PRIMARY KEY,
  plan_id                INT NOT NULL,
  day_number             INT NOT NULL,           -- 1–20
  title                  VARCHAR(200) NOT NULL,
  description            TEXT,
  estimated_time_minutes INT,
  activity_type          activity_type_enum,     -- 'guided' | 'semi_guided' | 'autonomous'
  skill_focus            VARCHAR(100)
);
```

## Activity types

<AccordionGroup>
  <Accordion title="guided">
    The AI or TL provides step-by-step instructions. Use this for new concepts where you need structure to avoid getting stuck. Typical duration: 30–45 minutes.
  </Accordion>
  <Accordion title="semi_guided">
    You receive a goal and hints, but must figure out the approach yourself. Builds problem-solving skills. Typical duration: 20–30 minutes.
  </Accordion>
  <Accordion title="autonomous">
    Open-ended challenge with no hints. Reserved for topics you've already practiced. Directly builds the `autonomy` soft skill dimension. Typical duration: 15–25 minutes.
  </Accordion>
</AccordionGroup>

## Plan lifecycle

- A plan is **active** when `is_active = true`. Only one plan per coder can be active at a time.
- When you request a new plan, all previous plans are set to `is_active = false` before the new one is generated.
- A plan spans **20 days** (days 1–20, organized in 4 weeks of 5 days each). Progress is tracked in `completed_days` as a JSONB map.
- After all 20 days are marked complete, the frontend shows a plan-complete banner prompting you to generate a new plan.

<Tip>
The `moodle_status_snapshot` and `soft_skills_snapshot` fields preserve a frozen copy of your data at the time the plan was generated. This lets TLs understand the context behind each plan even after your scores change.
</Tip>
